# Transcriptome Pyloneny 脚本整理规范

## 1. 目的
本规范用于统一整理项目中的各类脚本、依赖文件和说明文档。

后续所有目录的整理工作，默认都按照本文件执行，除非你另外提出新的特殊要求。

## 2. 适用范围
本规范适用于项目中的正式运行脚本，尤其是各流程目录中的 `.py` 文件。

不强制覆盖以下内容，除非后续明确要求处理：
- 第三方源码
- 历史备份脚本
- 教程示例
- 测试脚本
- 已归档旧版本

## 3. 目录整理规则
每个流程目录原则上应整理为下面这种结构：

```text
目录名/
├── 主运行脚本.py
├── README.md
└── dependencies/
    ├── 程序依赖文件
    ├── 配置模板
    ├── 数据库文件
    ├── adapter 文件
    └── 其他非正式运行脚本的依赖资源
```

具体要求如下：
- 正式运行的 `.py` 文件保留在流程目录的顶层。
- 如果一个流程目录中存在多个按顺序执行的小脚本，应优先新增一个主调度脚本作为唯一正式入口。
- 这种情况下，原有小脚本应移动到 `dependencies/` 中，作为依赖脚本由主调度脚本统一顺序调用。
- 除正式运行脚本之外，其他依赖文件统一放入 `dependencies/` 目录。
- `jar`、`fa`、`fasta`、模板、外部二进制辅助文件、说明性附件等，都优先归入 `dependencies/`。
- 如果一个目录中有多个正式运行脚本，则这些脚本都可以保留在顶层，其余依赖统一收纳。
- 非必要时，不随意改动项目根目录下的共享库目录，如 `lib/`。

## 4. 运行参数规范
所有正式运行脚本都应改为“脚本内配置参数”的模式，不再要求用户通过命令行传参。

统一要求如下：
- 不使用 `argparse`、`click` 等 CLI 参数输入方式作为主要运行方式。
- 所有需要用户修改的参数，都写在脚本最前面的配置区。
- 配置区应放在 import 之后、主逻辑之前。
- 配置项命名使用大写字母加下划线，例如 `INPUT_DIRECTORY`、`THREADS`。
- 输入目录和输出目录都必须作为独立参数写在脚本最前面的配置区。
- 每个正式脚本都应显式设置输入文件夹参数和输出文件夹参数，例如 `INPUT_DIRECTORY`、`OUTPUT_DIRECTORY`。
- 输入文件统一放在专门的输入文件夹中，不直接混放在脚本目录。
- 输出文件统一写入专门的输出文件夹中，不与输入文件混放。
- 用户需要填写的目录路径一律使用相对路径，不使用绝对路径。
- 路径类参数优先使用相对当前脚本目录的写法，避免写死外部绝对路径。
- 布尔参数使用 `True` 或 `False`，例如 `DRY_RUN = False`。

推荐格式如下：

```python
# ======================= CONFIGURATION =======================
INPUT_DIRECTORY = "input"
OUTPUT_DIRECTORY = "output"
THREADS = 20
DRY_RUN = False
# ============================================================
```

## 5. 脚本编写规范
整理后的正式运行脚本应尽量满足以下要求：
- 脚本可以直接通过 `python xxx.py` 运行。
- 用户只需要修改顶部配置区，不需要再拼接复杂命令。
- 尽量保留原有功能，不随意改变脚本核心逻辑。
- 对明显混乱的旧写法，可以做适度重构，但应以“更稳定、更清晰”为目标。
- 能保留的原输出文件命名规则尽量保留，避免下游流程被破坏。
- 如果脚本依赖项目共享工具函数，可以继续使用 `lib/` 中的公共函数。
- 路径处理优先使用 `os.path` 或 `pathlib`，避免依赖当前 shell 所在目录。
- 输入文件默认从配置区指定的输入目录读取。
- 输出文件默认写入配置区指定的输出目录；如果输出目录不存在，脚本应自动创建。
- 新增注释应简洁，只解释必要信息，不写过度冗长说明。

## 6. 依赖文件规范
依赖文件统一放入 `dependencies/` 目录后，应遵循以下规则：
- 正式运行脚本中引用依赖时，默认从当前脚本目录下的 `dependencies/` 查找。
- 依赖路径优先写成相对脚本目录的形式，例如：

```python
SCRIPT_DIR = os.path.dirname(os.path.abspath(__file__))
DEPENDENCY_DIR = os.path.join(SCRIPT_DIR, "dependencies")
```

- 如果依赖文件较多，可以在 `dependencies/` 下继续分子目录，如：
  - `dependencies/adapters/`
  - `dependencies/bin/`
  - `dependencies/db/`
  - `dependencies/templates/`

## 6.1 Conda 环境查找规范
对于依赖外部软件的正式运行脚本，统一增加一个 conda 环境相关参数，并写在脚本最前面的配置区。

推荐参数形式如下：

```python
CONDA_ENV_NAME = "trinity_env"
```

整理后的脚本应遵循以下查找顺序：
- 优先查找当前流程目录下 `dependencies/` 中是否已有对应软件
- 如果当前目录没有该依赖软件，再自动到指定的 conda 环境中查找
- 如果 conda 环境中也找不到，再尝试从当前系统 `PATH` 中查找
- 如果仍然找不到，脚本应明确报错，并告诉用户缺少哪个软件

推荐查找位置包括：
- `dependencies/bin/`
- `CONDA_PREFIX/bin/`
- `conda run -n 指定环境 which 软件名` 可定位到的路径
- 系统 `PATH`

脚本在实现时应尽量满足以下要求：
- 在脚本顶部写清楚默认使用哪个 conda 环境
- 不要求用户在运行前手动切换到对应环境
- 如果脚本需要调用 `java`、`Trinity`、`cd-hit-est`、`blastp`、`mafft` 等外部程序，应优先走自动查找逻辑
- 如果脚本能在当前目录依赖和 conda 环境之间自动切换，应在日志中打印最终实际使用的软件路径

说明：
- 这个规则主要针对“外部可执行程序”的查找，不强制用于 Python 包本身
- Python 第三方包是否安装在当前解释器中，仍由实际运行脚本时的 Python 环境决定

## 7. README 规范
每个整理过的流程目录都应有一个中文 `README.md`。

README 应至少包含以下内容：
- 该目录的功能简介
- 当前目录结构说明
- 哪个脚本是正式运行脚本
- `dependencies/` 中放了哪些依赖
- 需要用户修改哪些顶部参数
- 运行方式
- 重要注意事项

README 默认使用中文撰写，不再默认写成英文。

## 8. 验证规范
每次整理完一个目录后，至少进行以下检查：
- 检查目录结构是否符合规范
- 检查依赖文件是否移动到正确位置
- 检查脚本语法是否正常
- 确认脚本中的依赖路径已经同步更新
- 确认 README 内容与当前脚本实现一致

如果条件允许，优先执行：

```bash
python -m py_compile 主运行脚本.py
```

## 9. 命名约定
整理过程中默认遵循以下命名约定：
- 目录名保持原项目命名，不强制重命名
- 主运行脚本尽量保留原脚本名
- 依赖目录统一命名为 `dependencies`
- 文档统一命名为 `README.md`

## 10. 后续整理默认执行方式
后续你让我继续整理某个目录时，我默认按下面顺序处理：

1. 识别该目录中的正式运行脚本
2. 如果存在多个顺序执行的小脚本，先新增主调度脚本
3. 区分正式运行脚本与依赖文件
4. 新建或整理 `dependencies/`
5. 移动依赖文件并修正脚本中的引用路径
6. 去掉 CLI 参数输入，改为顶部配置区
7. 编写或更新中文 `README.md`
8. 做语法和结构校验

## 11. 需要单独确认的情况
如果后续遇到下面这些情况，我会单独和你确认再改：
- 一个目录里有多个脚本都可能是正式入口，且边界不清楚
- 移动文件可能影响下游脚本或外部调用
- 某些脚本实际上属于第三方源码，不适合重构
- 某些目录中既有正式流程，又有实验性脚本，难以直接分类

## 12. 当前默认约定
从现在开始，如果你让我继续整理某个目录，而没有额外说明，我默认：
- 按本规范执行
- README 使用中文
- 参数统一放到脚本最前面
- 非正式运行脚本和依赖资源尽量收纳到 `dependencies/`
- 正式脚本增加 conda 环境参数，并在本地依赖缺失时自动去该环境中查找外部软件
